Status: Phase 1 live on test address. Customer-facing DNS cutover in progress this week.
Built for: Clever Accounts & Workwell Accountancy — one codebase, two brands.
Pentest: External CREST-accredited engagement booked pre-broad-launch.

In one paragraph

A web app that gives every accountancy client a personal home page for everything to do with their account — where they are in onboarding, who their accountant is, how to book calls, where to sign documents, how to message us. One codebase serves both Clever Accounts and Workwell Accountancy — the app picks the right brand automatically based on which web address the client typed in. Hosted on Vercel in London. Backed by Salesforce as the source of truth, with a Supabase Postgres copy in London for fast reads. Clerk handles sign-in so we never see passwords. Built from day one to be tested by an outside security firm.

Reads (clients viewing their dashboard) hit the Postgres copy — fast.
Writes (Phase 2: clients uploading docs, sending messages) go straight to Salesforce.
Sync (Salesforce → Postgres): Salesforce broadcasts a change → our server picks it up → upserts into Postgres. Signed, idempotent, with a dead-letter queue if anything fails.

Security — designed for an outside pentest

We expect a CREST-accredited UK security firm to attack this before broad launch. Every layer below assumes the others might fail.

How clients are authenticated

  • Clients sign in with a passkey (Touch ID, Face ID, Windows Hello) or a magic link emailed to them
  • No passwords stored anywhere
  • Two-factor sign-in is optional now and will be required from broad launch
  • Sign-in handled by Clerk — they specialise in this so we don't have to roll our own password code
  • Sign-in cookies are HTTP-only, Secure-only, SameSite=Strict, and scoped to the portal subdomain only

How we prevent cross-account data leaks

This is the most important piece. Every data fetch in the app goes through one function called withPortalScope(). It:

  1. Reads the verified Clerk session
  2. Looks up which Salesforce Account that signed-in client belongs to
  3. Calls the rest of the code, passing only that Account's ID

It is structurally impossible for a route handler to accept a client-supplied "give me account X" parameter, because every data-fetching function only ever receives the Account ID derived from the verified session. Even a sloppily-written future page can't accidentally show one client's data to another.

As a second line of defence, the Postgres database has row-level security enabled — even if withPortalScope() were somehow bypassed, the database itself refuses to return rows that don't belong to the signed-in client.

How Salesforce and the portal trust each other

  • The portal uses an OAuth client-credentials flow to authenticate to Salesforce
  • Two separate Salesforce Connected Apps: one for the marketing site (creates Leads, low privilege), and a separate least-privilege one for the portal — it can only see records the portal needs. No salaries, no internal notes, no records belonging to clients other than the one being viewed
  • Messages from Salesforce to the portal are signed with an HMAC-SHA256 shared secret. Anything unsigned, with a bad signature, or with an old timestamp is rejected
  • The shared secret lives in Salesforce custom metadata and Vercel environment variables — never in the codebase

Browser and network defence

  • HTTPS only, with HSTS preload eligibility on the customer domain
  • Strict Content Security Policy on portal pages — locked-down allowlist of which third-party scripts can run (Clerk, Calendly, Supabase, Cloudflare Turnstile)
  • X-Frame-Options: DENY on portal — can't be embedded in another site (prevents click-jacking)
  • Referrer-Policy: same-origin — won't leak page URLs externally

Service abuse defence

  • Rate limits on every portal API endpoint via Upstash (30 req/min/user by default, tighter on sign-in)
  • Origin / Referer check on all POST endpoints — stops a malicious site posting on behalf of a signed-in user
  • Cloudflare Turnstile on the sign-up form before broad launch — invisible CAPTCHA that blocks bots without bothering humans

Audit trail

  • Every login, page view, action, and data fetch writes to a portal.audit_log table — actor, target, action, IP, user agent, timestamp
  • The table is append-only at the database level — no UPDATE or DELETE permissions for the portal service account
  • Will be shipped to a separate log aggregator (BetterStack or Axiom) for tamper-evident retention before broad launch

Operational hygiene

  • Production secrets stored only in Vercel encrypted environment variables — never in the codebase
  • .env.local is gitignored
  • Vercel Skew Protection is on — clients with warm tabs always hit their original deployment during rolling deploys
  • Vercel Firewall in front of every request — bot filtering, geographic rules, per-IP rate limits

Where data lives

All systems that hold client data are in UK or EU regions, confirmed at provisioning:

  • Vercel functions: London (lhr1)
  • Postgres: eu-west-2 (London)

What we deliberately chose NOT to build

  • In-house login. Clerk handles password hashing, breach detection, two-factor, recovery. We never see a password.
  • Browser-side Salesforce calls. Every Salesforce call goes through our server. The client browser never knows Salesforce credentials or that Salesforce is involved.
  • Microservices. Single Next.js app + Postgres. Fewer moving parts, fewer attack surfaces.
  • Custom email infrastructure. Resend + SendGrid handle delivery. SPF/DKIM/DMARC at the domain level.
  • Mobile app first. Web first, with a path to mobile via Capacitor reusing 100% of the backend.

Risks we've already engineered around

  1. Vendor lock-in. Salesforce stays canonical. The Postgres cache is rebuildable from Salesforce in minutes via the same sync pipeline that keeps it warm.
  2. GDPR data residency. Every system that touches client data is in UK or EU regions, verified at provisioning.
  3. GDPR right-to-erasure. Deleting a client in Salesforce cascades to the Postgres cache within seconds.
  4. Mobile future. Backend is API-first. A Capacitor wrap reuses 100% of the backend.
  5. Scale. The read-cache means we don't hit Salesforce's 100k/day API limit even with 6,500+ active clients.
